# 个人知识库建设建议

## 推荐路线

### 第一阶段：轻量可用

当前已经落地的是第一阶段：

```text
Markdown 文档 + graph.json + 离线 HTML 图谱
```

优点：

- 不依赖服务器。
- 可以直接用 Obsidian / VS Code 打开。
- 数据结构清楚，后续可迁移到 Web 应用或数据库。
- 适合先把工作经验沉淀起来。

### 第二阶段：增强展示

如果后续要做得更像产品，可以升级为：

```text
React / Vue + Cytoscape.js 或 Sigma.js + Markdown 搜索索引
```

能力：

- 更漂亮的图谱动画。
- 节点筛选、标签筛选、项目筛选。
- 点击节点直接显示 Markdown 内容。
- 支持全文搜索。

### 第三阶段：结构化知识系统

如果知识库规模很大，可以考虑：

```text
Neo4j / SQLite FTS / Meilisearch + Web 前端
```

适合：

- 项目很多。
- 技术、问题、设备、命令之间关系复杂。
- 需要跨项目检索和自动关联。

## 建议目录规范

```text
个人知识库/
  00_入口/          首页、索引、最近更新
  01_项目复盘/      每个项目一篇复盘
  02_技术地图/      技术概念卡片
  03_问题库/        问题、原因、解决方案
  04_设备与部署/    设备识别、部署命令、验收清单
  graph/            图谱页面和结构化数据
  assets/           图片、截图、附件
```

## 单篇项目复盘模板

```text
# 项目名称

## 一句话结论

## 我学到了什么

## 最终链路

## 是否满足目标

## 遇到的问题与解决

## 关键命令

## 后续优化

## 关联概念
```

## 单个问题卡片模板

```text
# 问题标题

## 现象

## 原因

## 解决

## 如何避免

## 关联项目
```

## 图谱数据规范

图谱数据存放在：

```text
graph/graph.json
```

每个节点建议包含：

```json
{
  "id": "唯一ID",
  "label": "展示名称",
  "type": "project/concept/library/issue/device/deployment/tool",
  "doc": "关联Markdown路径",
  "summary": "一句话摘要"
}
```

每条边建议包含：

```json
{
  "source": "起点ID",
  "target": "终点ID",
  "label": "关系说明"
}
```

## 我的推荐

当前先不要一上来做很重的知识库系统。建议先用这一版轻量结构持续沉淀 3 到 5 个项目。

当你发现以下问题时，再升级成完整 Web 知识库：

- 文档超过 100 篇。
- 搜索不够用了。
- 图谱节点超过 200 个。
- 你想要多人共享或浏览器部署。

这样不会在工具建设上过早消耗精力，而是先让知识真正沉淀下来。
